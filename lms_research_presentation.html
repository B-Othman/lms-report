Interoperability

Standards that matter in 2026

SCORM
1.2 → 2004 · Still the baseline
Standardizes content packaging (ZIP + imsmanifest.xml) and runtime tracking via JS API. Large legacy libraries depend on it. Implement SCORM 1.2 first; add 2004 sequencing on enterprise demand. Expect real-world quirks with suspend_data and branching.
ZIP packageRuntime APIimsmanifest
xAPI
Experience API · Requires LRS
Captures learning statements across contexts beyond LMS-launched e-learning — simulations, mobile, VR, coaching. Requires a Learning Record Store. Decide: embed LRS or integrate external LRS per tenant. Useful for richer analytics.
StatementsLRSCross-system
cmi5
xAPI profile · "xAPI with rules"
Adds session management and launch rules to xAPI for LMS-style import/launch/tracking. Combines SCORM's operational clarity with xAPI's richer data. Defines "cmi5 defined" vs "allowed" statements. Belongs after robust eventing is in place.
Launch rulesSession mgmtModern
LTI 1.3
Learning Tools Interoperability
Standardizes integration of external tools into LMS. LTI 1.3 uses OpenID Connect, signed JWTs, and OAuth 2.0. LTI Advantage adds NRPS, Deep Linking, and Grade Services. Critical for academic ecosystems; increasingly used in corporate.
OIDCJWTGrade passback
Identity standards: SAML 2.0 (enterprise SSO), OpenID Connect (modern auth on OAuth 2.0), SCIM RFC 7644 (lifecycle provisioning). TLS 1.3 for transport security.
05
Architecture

Multi-tenancy isolation models

Model Description Strengths Risks Best for
Pool (shared) One DB, shared tables + tenant_id Lowest infra cost, easiest ops, simple analytics Blast radius if data leaks; noisy-neighbor risk MVP, high-scale SMB
Silo (isolated) Separate DB / schema per tenant Strong isolation, per-tenant tuning, clean security story Ops complexity, migration burden, higher cost Regulated industries, large enterprise
Bridge (hybrid) Most tenants pooled; high-value tenants siloed Balances cost and isolation; supports tier pricing Requires mature routing + automation Mixed SMB + enterprise

Enforcement layers (pooled)

Application middleware — tenant context required on every request, injected automatically.
Database RLS — PostgreSQL Row Level Security, enabled per table, default-deny.
Test harness — automated cross-tenant data leak detection in CI.

White-labeling requirements

Custom domains / subdomains + managed TLS certificates per portal
Per-portal branding: logo, colors, typography, login pages, email templates
Portal-specific catalogs, certificates, and delegated admin roles

09
Pitfalls

Common mistakes & mitigations

Failure mode
Cross-tenant data leakage in shared-schema model
Mitigation
Enforce tenant context in middleware + DB-level RLS + automated cross-tenant test harness in CI
Failure mode
Underestimating SCORM variance across authoring tools and versions
Mitigation
Provide SCORM debug tooling; document supported versions; build compatibility test suites
Failure mode
Treating xAPI/cmi5 as "just another package type"
Mitigation
Implement or integrate LRS; build cmi5 session/state handling and statement validation
Failure mode
SSO added too late — enterprises won't pilot without it
Mitigation
Ship OIDC early; add SAML and SCIM as you move upmarket; follow OAuth security BCP
Failure mode
eCommerce without tax/receipt/audit readiness
Mitigation
Build orders/invoices/refunds as first-class entities with audit logs + payment/tax tooling
Failure mode
Accessibility bolted on later — procurement gates in edu/public sector
Mitigation
WCAG 2.2 baseline from MVP; keyboard nav, alt text, semantic structure across admin + learner UX

Universal recommendations: Start with a modular monolith (clear bounded contexts: identity, catalog, enrollments, learning runtime, reporting, billing) — evolve hot paths to services later. Use Kubernetes + HPA for autoscaling bursty LMS loads. Use PostgreSQL RLS as defense-in-depth for tenant isolation. Use object storage + CDN + adaptive streaming (HLS) for video delivery.
